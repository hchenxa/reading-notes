# Kafka学习笔记

## 入门
### 概述
#### 定义
- Kafka的传统定义: Kafka是一个**分布式**的基于**发布/订阅模式**的**消息队列**(Message Queue),主要应用于大数据实时数据处理领域。
- 发布/订阅：消息的发布按照类别进行分类，不会讲消息直接发送给特定的订阅者。订阅者只订阅自己感兴趣的消息。
- kafka最新定义: Kafka是一个开源的**分布式事件流平台**(Event Streaming),被广泛用于高性能**数据管道**,**流分析**,**数据集成**和**关键任务应用**。
#### 消息队列
常见的消息队列主要有kafka,ActiveMQ,RabbitMQ等。
- 传统消息队列的应用场景
  - 主要应用场景包括：缓冲/消峰，解耦和异步通信
  - 缓冲/消峰：有助于控制和优化数据流经过系统的速度，解决生产消息和消费消息的处理速度不一致的情况。
  - 解耦：允许独立的扩展或者修改两边的处理过程，只要确保他们遵循同样的接口约束。
  - 异步通信：允许用户把一个消息放入队列，但并不是立即处理它，等需要的时候再去处理它。 
- 消息队列的两种模式
  - 点对点模式
    - 一个生产者对应一个消费者,消费者消费完数据,数据在Queue里面删除
  - 发布和订阅模式
    - 消息队列里面有多个Topics
    - 消费者消费数据之后，数据不删除
    - 每个消费者相对独立，都可以消费到数据
#### 基础架构
- 一个topic可以分为多个partition,可以方便扩展,并提高吞吐量。
- 消费消息可以按照消费组的来进行消费，一个消费组里可以包含多个消费者，每个消费者并行消费消息。
- 一个partition里的数据只能由一个消费者进行消费。
- 为了提高可用性,每个partition可以增加副本,副本有leader和follower的区别,leader的副本用来数据处理,follower只有成为新的leader以后才能被生产和消费
- zookeeper: 用来记录kafka cluster一些信息,比如分区副本的leader,follower之类的

## 生产调优手册
### 硬件配置
1. 场景说明

100万日活,没人每天100条日志,每天总共的日志条数为100 * 100万 = 1亿条. 每秒需要处理1150条日志。每条日志大小为:0.5k ~ 2k,那每秒需要传输1150*1k - 1MB/s. 高峰期每秒数据量假设为20Mb/s。

2. 服务器数量
服务器数量（对应的经验值计算公式） = 2 * (生产者峰值生产速率 * 副本数 / 100) + 1

3. 磁盘的选择
kafka是按照顺序读写,所以需要考虑不同的硬盘顺序读写的速度差异。

1亿条数据 * 1k = 100G

100G * 2个副本 * 3天/0.7 = 1Tb

所以服务器的磁盘大小要大于1Tb

4. 内存选择
kafka内存分两部分:堆内存和页缓存

- 堆内存：(kafka内部配置) 10 - 15G. 启动参数里面的`-Xms` 和 `-Xmx`
- 页缓存：(服务器内存),只需要保证1个segment中25%的数据在内存中就可以，每个节点页缓存的大小 =（分区数 * 1g * 25%）/ 节点数。 例如10个分区，页缓存的大小 =（10 * 1g * 25%）/ 3（台服务器） = 1Gb左右

Note: 
- `jstat` 可以用来查看java进程的一些数据,比如可以通过`jstat -gc <pid> 1s 10`来查看gc了多少次.
- `jmap` 可以用来查看java进程的堆内存情况,比如可以通过`jmap -heap <pid>`来查看堆内存的使用情况

5. CPU选择
- `num.io.threads`: 负责写磁盘的线程数,整个参数值占总核数的50%.默认的io线程数是8.
- `num.replica.fetchers`: 副本拉取线程数,这个参数占总和数的50%的1/3.默认的线程数是1.
- `num.network.threads`: 数据传输线程数,这个参数占总核数的50%的2/3.默认的线程数是3.

6. 网络

选择千兆网卡就行

### 总体
1. 提高吞吐量
   1. 提高生产者吞吐量
      1. buffer.memory: 默认是32m，可以增加到64m
      2. batch.size: 默认是16k,如果太小，会导致频繁的网络请求,吞吐量下降;如果太大，会导致消息需要等待很久才能发出去，会增加网络延时.
      3. linger.ms: 默认是0,意思就是消息会立即被发送.一般设置在5-100毫秒。
      4. compression.type: 默认是none.使用压缩可以减小数据量，提高吞吐量。但是会加大CPU在生产端的消耗.
   2. 增加分区
   3. 提高消费者吞吐量
      1. fetch.max.bytes大小，默认是50m
      2. max.poll.records大小，默认是500条
   4. 增加下游消费者的消费能力
2. 数据精准一次
   1. 生产者角度
      1. acks设置为-1
      2. 幂等性(`enable.idempotence = true`) + 事务
   2. broker角度
      1. 分区副本大于等于2(`--replication-factor 2`)
      2. ISR里应答的最小副本数量大于等于2(`min.insync.replicas = 2`)
   3. 消费者角度
      1. 事务 + 手动提交offset(`enable.auto.commit = false`)
      2. 消费者输出的目的地必须支持事务(MySQL等)
3. 合理设置分区数
4. 单条日志大于1m
   1. `message.max.bytes`: 默认为1m,broker端接收每个批次消息的最大值
   2. `max.request.size`: 默认为1m,生产者发往broker每个请求消息最大值.针对topic级别设置消息体的大小
   3. `replica.fetch.max.bytes`: 默认为1m,副本同步数据,每个批次消息最大值
   4. `fetch.max.bytes`: 默认50m,消费者获取服务器一批消息最大的字节数.如果服务器端一批次的数据大于该值仍然可以拉取回来这批数据,因此,这不是一个绝对最大值.一批次的大小受`message.max.bytes`或者`max.message.bytes`的影响
